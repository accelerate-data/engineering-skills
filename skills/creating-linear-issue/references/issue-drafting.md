# Issue Drafting

Use this reference after discovery, field resolution, source-doc review, and any brainstorming, bug-intake, or decomposition handoff have returned.

## Draft Approval Content

Show the full draft before creating or updating the issue. Include:

- selected issue kind and why
- resolved project, milestone, assignee, and cycle
- selected-team User Flow label, rationale, and functional spec path when applicable
- related design docs read, or `not_applicable`
- brainstorming summary for non-trivial feature scope when applicable
- bug-intake summary for bug issues when applicable
- dedupe result
- issue draft outline
- decomposition approach when scope is too large for one issue

Create or update the Linear issue only after user approval.

## Issue Body Template

```md
## Problem
...

## Goal
...

## Non-goals
- ...

## Acceptance Criteria
- [ ] ...

## Risks
- ...

## Test Notes
- ...
```

For selected-team issues, include the functional spec path and related design doc paths in the description so implementation can trace source context.
