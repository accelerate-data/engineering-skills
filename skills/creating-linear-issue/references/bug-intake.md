# Bug Intake

Use this reference when the requested Linear issue is a bug, defect, regression, broken behavior, flaky behavior, or incorrect output.

## Intake Checks

| Check | Requirement |
|---|---|
| User language | Preserve the user's symptom and impact in plain language. |
| Expected vs actual | Capture what the user expected and what happened instead. |
| Reproduction | Record steps to reproduce when known; ask only if the issue cannot be filed durably without them. |
| Consistency | Note whether the issue is consistent, intermittent, environment-specific, or unknown. |
| Scope | Keep one root behavior per issue when possible. Split distinct symptoms or independent failure modes. |
| Codebase context | Search code/docs to learn domain language and user-facing boundaries. Do not turn the issue into a fix plan. |

## Bug Issue Shape

Bug issues should include:

- what happened
- expected behavior
- actual behavior
- impact or severity
- reproduction notes
- consistency or frequency
- acceptance criteria for proving the bug is fixed
